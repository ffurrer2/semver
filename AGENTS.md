# AGENTS Guide for `semver`

## Project shape (read this first)
- CLI entrypoint is `cmd/semver` (Cobra): each command lives in its own file and is wired in `init()` (example: `cmd/semver/compare.go`).
- Domain logic is in `pkg/semver` and should stay reusable/importable (no CLI dependencies).
- Shared CLI plumbing is in `internal/pkg/cli` (`Apply` for stream-per-item, `Map` for batch); both call `os.Exit(0)` internally.
- Small utility packages support domain code: `internal/pkg/number` (numeric parsing/comparison), `internal/pkg/predicate` (functional combinators).
- Build metadata/version info comes from `internal/pkg/app/version.go`, with linker injection from Task/GoReleaser ldflags.

## Runtime/data flow conventions
- Commands that require valid semver values parse inputs via `semver.Parse(...)`, print errors with `cmd.PrintErrf("error: %v\n", err)`, then `os.Exit(1)` (examples: `cmd/semver/compare.go`, `cmd/semver/format.go`, `cmd/semver/major.go`).
- `validate` is the intentional exception: it uses `semver.IsValid(...)` and exits `1` on first invalid input (`cmd/semver/validate.go`).
- Commands accepting variadic input follow this pattern:
  - CLI args present -> process args
  - no args -> read stdin line-by-line (`cli.Apply` / `cli.Map`)
- `sort` is the batch command pattern (`cli.Map` + `sort.Sort(semver.BySemVer(...))` in `cmd/semver/sort.go`).
- `filter` composes predicates (`predicate.Or(...)`) instead of nested if-chains (`cmd/semver/filter.go`).
- `format` compiles templates once with `template.New(...).Funcs(sprig.FuncMap())` and executes them per item against `Major/Minor/Patch/PreRelease/BuildMetadata` fields (`cmd/semver/format.go`).

## SemVer behavior that must not regress
- Parsing/validation is regex-based (`pkg/semver/semver.go`, `NamedGroupsPattern`), then converted via named groups.
- `CompareTo` compares major/minor/patch + prerelease precedence; build metadata does not affect ordering.
- `NextMajor/NextMinor/NextPatch` strip prerelease/build metadata if current version is prerelease (do not increment numeric parts in that case).
- Overflow in `Next*` panics with exported errors (`ErrNextMajorUintOverflow`, etc.); tests rely on this behavior.

## Build, test, lint workflows
- Prefer Task targets over ad-hoc commands:
  - `task build` -> static build to `artifacts/bin` with ldflags from `tasks/GoTasks.yml`
  - `task test` -> `go test -race -covermode atomic ./...` + HTML coverage + container structure test
  - `task lint` -> Go/Markdown/YAML lint stack
  - `task scan` -> Dockle + Trivy image scans against `ghcr.io/ffurrer2/semver:latest`
  - `task goreleaser:build` -> local GoReleaser build validation (`--single-target --snapshot`)
  - `task goreleaser:check`, `task goreleaser:snapshot` for packaging flows
- Version strings are generated by `scripts/version` (tag/branch/dirty-aware semver-like output).

## Testing patterns
- Tests use Ginkgo/Gomega (`*_suite_test.go`, `Describe`/`DescribeTable` style), not plain table tests only.
- High-value tests are extensive in `pkg/semver/semver_test.go`; mirror its style when adding behavior.

## Release and integration boundaries
- Release config is `build/package/.goreleaser.yaml` (archives, Homebrew cask, Scoop, Docker Hub, GHCR).
- Container quality gates use `test/semver_container_test.yml` and CI scanners (Trivy/Anchore in `.github/workflows/ci.yml`).
- External CLI/runtime dependencies to be aware of: `spf13/cobra`, `go-task/slim-sprig`, `samber/lo`.

## Repo-specific guardrails for agents
- Keep SPDX header (`// SPDX-License-Identifier: MIT` or YAML equivalent) in new source/config files.
- Keep imports and formatting aligned with project tooling (`gofumpt`, `goimports`, `golangci-lint` config in `.github/linters/.golangci.yml`).
